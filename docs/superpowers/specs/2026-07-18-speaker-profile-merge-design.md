# Слияние профилей спикеров + подстановка имён (merge/move + naming)

- **Задача:** vts-552
- **Дата:** 2026-07-18
- **Зависит от:** vts-80i (реестр голосов — closed), vts-5xz (диаризация — closed)
- **Источник требований:** Linear VOS-65

## Термины

- **Голос** — участник диалога, распознанный в задаче. Технически `speaker_label`
  (`SPEAKER_00`) внутри задачи; своей строки в БД нет, живёт в `diarization.json`
  и в `MatchDecision.speaker_label`.
- **Персона** — участник, сохранённый в реестре. Модель `Speaker`.
- **Фрагмент** — сохранённый кусок голосовых данных для матчинга. Модель `VoiceSample`.
  Содержит И эмбеддинг (`Vector(256)`, для матчинга), И аудио-байты (послушать), И
  `source_task_id`. **Фрагмент ≠ эмбеддинг:** эмбеддинг — одно из полей фрагмента.
  merge/move двигают весь `VoiceSample` целиком.

## Связи (фундамент — легко перепутать)

В схеме vts-80i НЕТ цепочки «голос → фрагмент → персона». Есть две связи:

1. **Фрагмент → персона:** `VoiceSample.speaker_id` — кому принадлежит отпечаток.
   Источник для БУДУЩЕГО матчинга (`nearest_speakers` сравнивает новые голоса с
   фрагментами и по `speaker_id` понимает, чей отпечаток).
2. **Голос → {персона, фрагмент}:** `MatchDecision` = `(speaker_label, speaker_id,
   voice_sample_id, source_task_id, outcome, distance)` — снимок решения «в задаче X
   голос SPEAKER_00 привязан к персоне Вася по фрагменту F». Источник привязки
   «голос → персона» ДЛЯ КОНКРЕТНОЙ ЗАДАЧИ, и одновременно калибровочная статистика.

Привязка голоса к персоне в рамках задачи берётся из **решения** (`MatchDecision`), а
не из фрагмента. Фрагмент — источник для будущих матчей, не для исторических имён.

## Операции и влияние на данные

| Операция | Что меняется | `MatchDecision` | Статус |
|---|---|---|---|
| Удалить фрагмент | `VoiceSample` удаляется | `voice_sample_id → SET NULL` (авто) | ✅ vts-80i |
| Удалить персону | все её `VoiceSample` + `Speaker` | `speaker_id → SET NULL` (авто) | ✅ vts-80i |
| Переместить фрагмент | `VoiceSample.speaker_id` → другая персона | **не трогаем** | ❌ новое |
| Слить персоны | все `VoiceSample` источника → цель; источник удалён | `speaker_id`: источник → цель (**переписываем**) | ❌ новое |

**Ключевое различие move vs merge — разные правила для `MatchDecision`, и это не
противоречие, а различение:**

- **Move фрагмента** — калибровочная операция над одним отпечатком. Исторические
  решения НЕ трогаем: `MatchDecision` хранит «при какой дистанции человек
  подтвердил/отверг», и переписывать это задним числом — исказить калибровку.
- **Merge персон** — утверждение «это один человек». Переписываем
  `MatchDecision.speaker_id` источника на цель. Калибровочный смысл не страдает:
  `outcome` и `distance` не меняются, меняется только к какой персоне отнесён
  результат — а персоны и так объявлены одним человеком. Это НЕ подделка решения.

Без переписывания (`SET NULL` при удалении источника) старые задачи, где голос был
привязан к персоне-источнику, потеряли бы имя — неверный исход: пользователь слил
Васю-1 в Васю-2, ожидая «Вася» везде, а старая задача показала бы «Голос N».

**Важно для имплементации — удаление и merge удаляют персону по-РАЗНЫМ путям:**
- **Удаление персоны** (существует): `MatchDecision.speaker_id → SET NULL` (FK, авто) —
  голоса становятся анонимными, это ЖЕЛАЕМЫЙ исход (пользователь удалил персону).
- **Merge**: ПЕРЕД удалением источника переписываем `MatchDecision.speaker_id` на цель,
  так что к моменту `delete Speaker` решений, ссылающихся на источник, уже нет — SET NULL
  ничего не обнуляет. Порядок шагов в `merge_speakers` (переписать решения → потом
  удалить источник) гарантирует, что имена не теряются. Перепутать порядок = потерять
  имена в старых задачах.

**Merge = «переместить ВСЕ фрагменты источника в цель» + «переписать все решения
источника на цель» + «удалить источник».** Move — примитив; merge построен на нём.
Одна репо-операция `reassign_speaker_samples(from, to)` — основа обеих.

**`source_task_id` при move/merge — сохраняем.** Фрагмент переезжает к другой персоне,
но «из какой задачи родом» не меняется — это факт о происхождении звука, не о
принадлежности. FK уже `SET NULL`, висячих ссылок не будет.

**Дедупликацию фрагментов при merge не делаем.** После merge у цели может оказаться
два фрагмента от одной задачи для одного `speaker_label` — не проблема: max-агрегация
(`MIN(distance)`) берёт ближайший, лишний только улучшает покрытие («больше фрагментов
— лучше матчинг», vts-80i). YAGNI.

## Репозиторий и API

Три репо-операции, все user-scoped (граница безопасности), все `flush()` не `commit()`:

```
move_voice_sample(user_id, sample_id, target_speaker_id) -> VoiceSample | None
    # фрагмент И цель принадлежат user_id; VoiceSample.speaker_id = target
    # MatchDecision не трогаем; None если не найдены/чужие

merge_speakers(user_id, source_id, target_id) -> bool
    # source != target, оба user_id; в одной транзакции:
    #   1. VoiceSample источника: speaker_id = target
    #   2. MatchDecision где speaker_id=source: speaker_id=target
    #   3. delete Speaker источника
    # False если не найдены/чужие/source==target

reassign_speaker_samples(user_id, source_id, target_id)  # примитив под обе
```

**Эндпоинты** (по образцу `/api/speakers/*` из vts-80i):

```
POST /api/speakers/{speaker_id}/samples/{sample_id}/move  {target_speaker_id} -> 200/404
POST /api/speakers/{source_id}/merge                      {target_id}         -> 204/404/409(source==target)
```

Обе — **одна транзакция, один commit в конце**. Для merge критично: перенос
фрагментов + переписывание решений + удаление источника атомарны, иначе частичный сбой
оставит реестр в противоречии (фрагменты переехали, решения нет; или источник удалён с
осиротевшими фрагментами). Та же транзакционная граница, что в эндпоинте разрешения
голосов.

Move к «новой персоне»: создание `Speaker` + перенос фрагмента в одной транзакции —
чтобы не осталась пустая персона при сбое.

## Подстановка имён при рендере

Два куска, разделённых по стоимости.

### Сырой транскрипт — при рендере (бесплатно, автоматически)

Сейчас `label_map` (merge.py:550) печатает `SPEAKER_00 → «Голос 1:»` (порядковый
номер). Меняем: сматченный голос → имя персоны. Цепочка:

1. `speaker_label` из сегмента,
2. `MatchDecision` этой задачи с этим `speaker_label` → `speaker_id`,
3. текущее `Speaker.name`.

Не сматчен (`speaker_id` NULL — промах или персона удалена) → откат на «Голос N».
Подстановка **аддитивна**: где имени нет, поведение не меняется. Работает
автоматически после merge: `MatchDecision.speaker_id` переписан на цель, `Speaker.name`
берётся живой при рендере — без пересчёта.

**Язык метки:** имя персоны — собственное, НЕ переводится («Вася:» в любом языке).
Подстановка имени обходит языковую логику `speaker_label_word` (vts-5xz). Сплиттер
утверждений (`_UTTERANCE_RE`) должен продолжать узнавать И именованные метки («Вася:»),
не только словарные «Голос N:»/«Speaker N:» — иначе диалоговый режим для именованных
транскриптов молча сломается.

### Обработанный транскрипт и саммари — только при перегенерации (прогон LLM)

Механизм уже есть — `restart_summary` (`vts/api/main.py`), режимы `full` /
`final_only` (`RestartSummaryRequest`). Строить не нужно. Нужно, чтобы путь генерации
получал **список участников диалога** (см. ниже) и подставлял имена.

- `full` — пересобирает обработанный транскрипт с именами, потом суммаризирует;
- `final_only` — по уже обработанному транскрипту (старый, с «Голос N» → имён не
  будет; хочешь имена — `full`).

Merge под то же правило, что rename в vts-80i: саммари обновляется только явной
перегенерацией, UI уже предупреждает. Автоперегенерации при merge нет.

### Список участников диалога в промпте (ключевое улучшение)

Вместо цитирования конкретной метки («сохрани "Голос N:"» — контракт `_keep_speakers_
instruction`, vts-5xz) промпт получает **список участников**, собранный из данных
задачи, двухчастный:

> Имена участников диалога: Вася, Петя, Таня.
> Также в диалоге присутствуют участники с неизвестными именами: Голос 2, Голос 4.

Почему это лучше простого сохранения меток:

- модель заранее знает ВЕСЬ набор говорящих (именованные + анонимные), так что смесь
  «Вася:» + «Голос 4:» перестаёт быть неожиданностью — контракт «инструкция описывает
  то, что в тексте» (vts-5xz) выполняется строже, а не ломается;
- **чинит имена whisper.** Whisper распознаёт имена на слух и коверкает («Вася» →
  «Ваня»/«Вазя»); зная список реальных участников, модель может исправить их в тексте.
  Поэтому список даётся И на обработке транскрипта (где текст ближе к исходному), И на
  финальной суммаризации, а не только в саммари.

Источник списка — та же цепочка: именованные из `MatchDecision → Speaker.name`,
анонимные из несматченных `speaker_label → «Голос N»`.

**Передача — через две переменные шаблона `${NAMED_SPEAKERS}` и `${ANONYMOUS_SPEAKERS}`**,
по образцу существующих `${LANG}` / `${INPUT_WORDS}` / `${TARGET_WORDS}` (подстановка
`prompt.replace("${VAR}", value)` в `render_prompt_*`, `summarization.py`). Формулировка
живёт В ПРОМПТЕ (`segment_prompt.md` + финальный), редактируется без правки кода; код
только вычисляет значения.

**Переменная содержит ГОТОВУЮ фразу целиком (или пустую строку), не голый список** — как
`${LANG}` подставляется готовым значением, а не куском под обёртку. Код собирает:
- `${NAMED_SPEAKERS}` = «Имена участников диалога: Вася, Петя, Таня.» или `""`;
- `${ANONYMOUS_SPEAKERS}` = «Также в диалоге присутствуют участники с неизвестными
  именами: Голос 2, Голос 4.» или `""`.

Так недиаризованная/безымянная задача не оставляет висячих заголовков («Имена
участников:» без имён): переменная просто пуста, фраза исчезает. Промпт-файл содержит
`${NAMED_SPEAKERS}` и `${ANONYMOUS_SPEAKERS}` в нужном месте. Заменяет
`_keep_speakers_instruction`. Формулировка — в языке вывода (`${LANG}`), собирается с
учётом языка записи (как `speaker_label_word`).

Поскольку список — часть текста для LLM, имена в саммари/обработанном транскрипте
появляются только при перегенерации (согласуется с разделением по стоимости).

## UI

Всё встраивается в существующий диалог реестра голосов (vts-80i, задача 13). Новых
диалогов нет. DOM-блоки — до тега `app.js` (правило без `defer`). i18n ru/en/de.

### Move фрагмента (правая колонка — фрагменты персоны)

- Кнопка «переместить» у каждого фрагмента (рядом с «удалить»).
- Дропдаун (переиспользуем компонент из диалога разрешения голосов, vts-80i задача 14):
  «\<Создать новую персону\>» ПЕРВЫМ пунктом, далее персоны **по близости эмбеддинга
  фрагмента** (по умолчанию), с переключателем на алфавит. «Создать новую» первым в
  обоих режимах. Близость: у фрагмента есть эмбеддинг, `nearest_speakers` готов — move
  почти всегда исправление («диаризация раздробила Васю, переношу к нему»), и «на кого
  похож» — ровно тот вопрос. Кап кандидатов — `speaker_match_candidates_cap` (vts-4rt).
- Подтверждение: «Фрагмент будет перемещён к персоне «Петя». Продолжить?»
- После move — правая колонка обновляется.

### Merge персон (левая колонка — персоны)

- Кнопка «слить» у каждой персоны (рядом с «переименовать»/«удалить»). Кнопка стоит у
  персоны-ИСТОЧНИКА (исчезнет), выбираем ЦЕЛЬ (останется).
- Список персон-целей: та же дропдаун-логика, сортировка **по алфавиту** (близость
  персоны к персоне посчитать не из чего).
- Подтверждение, явный однонаправленный текст: «Все голосовые данные персоны
  **«Вася-1»** будут перенесены в **«Вася-2»**. Персона **«Вася-1»** будет удалена.
  Продолжить?» — источник и цель по именам, судьба каждой проговорена. Слияние
  необратимо и удаляет данные, поэтому текст, а не значок «→» (проектный UI-паттерн).
  Переименование объединённой персоны НЕ предлагаем — операция rename уже есть отдельно.
- После merge — левая колонка обновляется; если была выбрана слитая — переключаемся на
  цель.

### Матрица подтверждений (весь диалог реестра)

| Операция | Подтверждение |
|---|---|
| Удалить персону | «удалить X и N фрагментов?» (vts-80i) |
| Удалить фрагмент | «удалить фрагмент?» (vts-80i) |
| Переместить фрагмент | «переместить к «Петя»?» (новое) |
| Слить персоны | «данные «Вася-1» → «Вася-2», «Вася-1» удалится?» (новое) |

Добавление фрагмента в реестре не предусмотрено — только из задач (vts-80i).

## Конфиг

Новых настроек нет. Move-дропдаун переиспользует `speaker_match_candidates_cap`
(vts-4rt).

## Тесты

- **move:** `VoiceSample.speaker_id` меняется, `MatchDecision` не трогается; изоляция
  (чужой фрагмент/цель → 404); move к «новой персоне» создаёт её и переносит в одной
  транзакции;
- **merge:** все фрагменты источника → цель; все `MatchDecision.speaker_id` источника →
  цель; источник удалён; `source == target` → 409; изоляция → 404;
- **транзакционность merge:** сбой на середине не оставляет частичного состояния;
- **подстановка имён (рендер транскрипта):** сматченный голос → имя; несматченный →
  «Голос N»; после merge старая задача показывает объединённое имя; удалённая персона →
  откат на «Голос N»;
- **сплиттер утверждений** узнаёт именованные метки («Вася:»), не только «Голос N:»;
- **список участников:** `${NAMED_SPEAKERS}` / `${ANONYMOUS_SPEAKERS}` собраны как
  готовые фразы из сматченных (имена) + несматченных (анонимные); пустые строки для
  недиаризованной/безымянной задачи (нет висячих заголовков); подставляются в обработке
  и финале;
- **перегенерация** (`restart_summary` `full`) собирает вход со списком участников;
- **UI (verifier-web):** кнопки move/merge, дропдаун с переключателем сортировки, оба
  подтверждения с правильным текстом направления.

## Границы (не входит)

- Дедупликация фрагментов при merge — YAGNI.
- Undo merge — merge необратим; защита через подтверждение, не через откат.
- Автоперегенерация саммари при merge — только по кнопке (как rename, vts-80i).
- Алиасы персон — отвергнуто в пользу переписывания решений.
- Множественный выбор персон в UI — отвергнуто в пользу кнопки-у-персоны (консистентнее;
  множественный выбор нигде не используется).

## Проверить при имплементации

- `_UTTERANCE_RE` (merge.py) — матчит ли произвольное имя перед двоеточием, или только
  словарные «Голос»/«Speaker»; если только словарные — расширить, иначе именованный
  диалог не разобьётся на реплики.
- Где именно вставляется `label_map` в пути генерации обработанного транскрипта — чтобы
  подстановка имён и список участников легли туда же, а не в двух местах.
- Точка подстановки `${NAMED_SPEAKERS}` / `${ANONYMOUS_SPEAKERS}` — в `render_prompt_*`
  рядом с `${LANG}`, заменяя `_keep_speakers_instruction`; проверить оба пути (основной и
  overflow-re-chunk из vts-5xz). Добавить переменные в `segment_prompt.md` и финальный
  промпт.
