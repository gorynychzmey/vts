# Диаризация транскрипций (vts-5xz)

**Дата:** 2026-07-15
**bd:** vts-5xz
**Linear:** [VOS-65](https://linear.app/vostrikov/issue/VOS-65)
**Тип:** новая функциональность
**Связано:** vts-80i (именование спикеров + enrollment) — блокируется этой задачей

## Цель

Размечать транскрипт по спикерам и автоматически выбирать режим рендера:
плоский текст для монолога, «Голос N: …» для диалога.

Основной сценарий пользователя — рабочие видеовстречи на 4+ участников, где
перебивания частотны. Это определило ключевые решения ниже.

## Обнаруженные факты (обоснование дизайна)

Исходный дизайн задачи (2026-05) опирался на предпосылки, которые проверка кода
и ресёрч опровергли. Зафиксировано, чтобы решения не пересматривались вслепую.

### PyTorch в проекте НЕТ

Исходный дизайн: «speechbrain тянет PyTorch — у нас уже есть для whisper».
Неверно. `requirements.txt` содержит только веб-стек и `httpx`. Whisper — чужой
контейнер за HTTP (`whisper:9000`, образ `ahmetoner/whisper-asr-webservice`),
клиент — `vts/services/transcription/_base.py`. **VTS это оркестратор; ML-модели
живут в отдельных контейнерах.**

Следствие: диаризация НЕ идёт зависимостью в основное приложение (иначе +PyTorch
~2-3 GB в образ, который его не имеет). Идёт отдельным контейнером за HTTP.
Побочный плюс: выбор библиотеки становится деталью ВНУТРИ контейнера, цена
ошибки в стеке резко падает.

### AsrSegment — не фраза, а кусок аудио

Исходный дизайн (и первая редакция этой задачи в bd) предполагал колонку
`speaker_label` в `asr_segments`. **Не годится.** `AsrSegment`
(`vts/db/models.py:197`) — это крупный кусок аудио от `SegmentAudioStep`,
нарезанный по длительности для параллельной транскрипции; в `text` лежит текст
всего куска. Внутри одного куска говорят несколько человек. Спикер меняется на
порядок чаще гранулярности этой таблицы.

Настоящие фразы с таймкодами: `AsrSegment.raw_json` (verbose_json от Whisper) и
`transcript.json → entries` (собирается `MergeTranscriptStep` из start/end/text).

### Word-level таймкоды УЖЕ приезжают

`_asr.py:20` запрашивает `word_timestamps: "true"`. Слова с таймкодами уже лежат
в `raw_json` — `MergeTranscriptStep` их просто выбрасывает, собирая `entries`
только из start/end/text. Точный мёрж не требует менять бэкенд.

`cpp` шлёт только `response_format: verbose_json`, слов не просит. Боевой
бэкенд — `asr` (`config.yaml:16`); `cpp` — наследие.

### Стек: две из четырёх причин отказа 2026-05 больше не действуют

- **pyannote community-1 НЕ gated.** Проверено скачиванием весов анонимно, без
  токена: `segmentation/pytorch_model.bin` → HTTP 200, 5 906 507 B;
  `embedding` → 26 646 242 B; sha256 совпал с заявленными LFS OID. Нюанс:
  официальный репо `pyannote/` — `gated: auto` (401 анонимно), зеркало
  `pyannote-community/` — `gated: false`. README зеркала ВЫГЛЯДИТ gated: это
  дословная копия с унаследованным неактивным `extra_gated_prompt` frontmatter.
  **Карточке модели тут верить нельзя.** Код MIT, веса CC-BY-4.0 (обязательство
  атрибуции, которого нет у Apache-2.0).
- **NeMo:** появился non-gated CC-BY-4.0 `diar_streaming_sortformer_4spk-v2`
  (раньше NC). Но «слишком тяжёлый» в силе: `nemo_toolkit[asr]` = 60
  зависимостей (lightning, wandb, optuna, телеметрия) + жёсткий пин
  `transformers~=4.57.0`. Отклонено повторно. Осторожно: родственные модели имеют
  ТРИ разные лицензии (v2.1 — NVIDIA Open Model License, вопреки саммари карточки).
- **DiariZen:** код MIT, веса CC-BY-**NC** → дисквалифицировано.
- **speechbrain ECAPA:** остаётся валидным (Apache-2.0, не gated, EER 0.80%).
  Показательно: сам pyannote использует ECAPA как эмбеддер.

**Почему всё же pyannote:** spectral clustering (исходный выбор) принципиально не
умеет overlapping speech — модель «один спикер на фрейм». Это не тюнится.
pyannote 3.1+ решает powerset-сегментацией. Отстаёт не ECAPA, а clustering-only
пайплайн. Для встреч 4+ с перебиваниями overlap — основной режим, а не краевой
случай.

**Ловушка в бенчмарках:** AMI у speechbrain (2.43%/4.03%) — collar 0.25s и
ИГНОРИРОВАНИЕ overlap; 17.0% у pyannote — без collar, с overlap. Числа
НЕсравнимы; наивное чтение таблиц уводит решение не туда.

**Риск цепочки поставок:** зеркало ведёт один непроверенный пользователь. Веса
вендорим по проверенным sha256, не тянем из HF в рантайме. Fallback:
`pyannote/wespeaker-voxceleb-resnet34-LM` качается анонимно от first-party —
тогда от зеркала зависит только segmentation.

## Принятые решения

- **Контейнер за HTTP, не библиотека.** PyTorch не появляется в
  `requirements.txt` основного приложения.
- **Атрибуция — структурированно, не в БД.** Требование VOS-65 «привязка
  фраза→спикер хранится в модели данных, а не только в рендере текста» читается
  как «структурированно» (решение Виктора, 2026-07-15): `speaker` в
  `transcript.json → entries`. Новая таблица фраз НЕ заводится — потребитель
  БД-запросов появится только в vts-80i, тогда и решать, зная реальный матчинг.
- **Точный мёрж с деградацией, не два алгоритма.** Максимальное перекрытие —
  базовый уровень; при наличии слов включается разрезание фраз. `cpp` тихо
  деградирует до перекрытия (наследие, не в ходу).
- **Эмбеддинги отдаются сразу**, хотя vts-5xz ими не пользуется — pyannote 4.x
  возвращает их из той же диаризации бесплатно (`output.speaker_embeddings`), а
  vts-80i без них переобсчитывал бы аудио. Класть в контракт потом — переделывать
  контейнер и клиент.
  (README pyannote утверждает, что эмбеддингов нет — README устарел, проверено по
  исходникам; есть и `speaker_verification.py`.)
- **Дефолт `diarize: false`.** Новая фича на неоткалиброванных порогах + лишний
  проход по аудио. Включить потом легко, откатить — сложнее.

## Архитектура

### Контейнер

Сервис в `docker-compose.yml`, профиль `diarize` (по аналогии с `asr-whisper`),
внутри pyannote 4.x. Веса предзагружаются при сборке образа по проверенным
sha256 — без рантайм-обращений к HF.

### Клиент

`vts/services/diarization/` зеркалит `vts/services/transcription/`:

```
__init__.py    # create_diarization_backend(url, backend)
_base.py       # DiarizationBackend(ABC): .diarize(audio_path) -> dict
_pyannote.py   # PyannoteBackend — httpx, по образцу _asr.py
```

`_base.py` переиспользует приём `WhisperBackend._post_audio`: multipart POST
аудио, `raise_for_status`, нормализация ответа.

### Контракт

`POST /diarize`, multipart с аудиофайлом:

```json
{
  "segments": [{"start": 0.0, "end": 12.3, "speaker": "SPEAKER_00"}],
  "embeddings": {"SPEAKER_00": [0.12, "…192-dim"]},
  "num_speakers": 3
}
```

## Пайплайн

Новый шаг `diarize` в `STEP_REGISTRY`, **между** `transcribe_segments` и
`merge_transcript`.

**Диаризуется целое аудио** (`audio.wav`), НЕ куски из `SegmentAudioStep`. Куски
режутся по длительности для параллельной транскрипции; один человек в разных
кусках получил бы разные метки — диаризация не знает, что это тот же голос. По
целому файлу метки сквозные.

Шаг пишет `outputs/diarization.json` (segments + embeddings + num_speakers).
`already_done` — через существование файла (как у остальных шагов), то есть
переживает рестарт воркера бесплатно.

`MergeTranscriptStep` при сборке `entries` приписывает спикера. **Нет
`diarization.json` (диаризация выключена) → работает ровно как сейчас**, ни одна
ветка не меняется.

## Мёрж: два уровня точности

Один алгоритм с деградацией:

1. Для фразы берём слова из `raw_json` (у `asr` уже есть).
2. Каждому слову — спикер по перекрытию с сегментами диаризации.
3. Группируем подряд идущие слова одного спикера.
4. **Группа отделяется в свою entry только если ≥2 слов И ≥0.8 сек** (оба
   условия). Не прошедшие порог поглощаются **предыдущей** группой.
5. Нет слов в `raw_json` (`cpp`) → откат: вся фраза одному спикеру по
   максимальному перекрытию.

**Почему порог (шаг 4).** Смена спикера внутри фразы бывает двух сортов:
настоящая смена реплики («…да, согласен. — А ты что думаешь?») и короткий вброс
поверх («угу», «да-да»). На встречах 4+ фоновые поддакивания — основной источник
шума. Резать чужую фразу на три части из-за «угу» — искрошить текст в лапшу. При
этом у диаризации на коротких сегментах точность заведомо низкая: 300-мс «угу» —
ровно тот случай, где эмбеддинг ненадёжен. Резать текст по неуверенному сигналу —
плохая сделка. Терять «угу» не жалко: смысла нет, стоимость ошибки высокая.

**Почему поглощает предыдущая, а не ближайшая по времени.** «Угу» посреди чужой
фразы прилипает к тому, кто говорил до него, и текст не переставляется.

Пороги — в конфиг, откалибровать на реальной встрече.

## Рендер и выбор режима

1. Отбрасываем спикеров с долей речи **< 5%** — гасит ложного спикера,
   выдуманного на шуме/музыке/эхе.
2. Остался один → **плоский текст**, как сейчас.
3. Осталось несколько → **диалог**: `Голос 1: …`, переключение на смене.

**Переформулировано против исходной задачи.** Там было «один спикер доминирует
≥95% времени → монолог». Критерий завязан на долю времени, тогда как надёжнее
смотреть на ЧИСЛО спикеров: нашли одного — монолог, порог не нужен. Порог нужен
для другого — гасить ложного второго спикера. Это разные вопросы; 95% как
«выбор режима рендера» для встреч 4+ не сработал бы почти никогда.

Метки `SPEAKER_00` → `Голос 1..N` по порядку **первого появления**: «Голос 1» —
тот, кто заговорил первым.

**В `entries` кладём ТЕХНИЧЕСКУЮ метку** (`SPEAKER_00`); `Голос N` — только
рендер. Иначе vts-80i не сможет переименовать спикера, не переписывая артефакт.

Транскрипт отдаётся плоским `.txt` (`transcript_path`, постранично через API) →
`Голос 1:` пишется в текст при сборке `transcript.txt`. **UI менять не нужно.**

## Опции и конфиг

`core/config`:
- `diarization_url`
- `diarization_backend: str = "pyannote"`
- `diarization_enabled_default: bool = False`
- пороги мёржа (≥2 слов, ≥0.8 сек) и отсечка спикера (<5%)

Pipeline options / `submit_video`: `diarize: bool | None` (`None` → из конфига).

`vts/api/schemas.py`: `diarize` рядом с `transcript`; по образцу
`prompts require transcript` — валидация, что `diarize` требует `transcript`.

## Тестирование

- Мок бэкенда диаризации (HTTP не дёргаем) — по образцу тестов транскрипции.
- **Мёрж-логика** — основное:
  - фраза целиком внутри одного сегмента спикера;
  - фраза накрывает смену спикера → режется на две entry;
  - короткий вброс («угу») ниже порога → поглощается предыдущей группой, entry
    не появляется;
  - вброс выше порога → отделяется;
  - нет слов в `raw_json` → откат на максимальное перекрытие.
- **Выбор режима:** один спикер → плоский; несколько → диалог; ложный спикер с
  долей <5% → отброшен, режим плоский.
- **Регрессия:** нет `diarization.json` → `entries` и `transcript.txt`
  байт-в-байт как сейчас.
- Маппинг меток по первому появлению.

## Открытые вопросы (в имплементацию, не блокируют)

- Пороги (≥2 слов, ≥0.8 сек, <5%) — прикидки, калибровать на реальной встрече 4+.
- Атрибуция CC-BY-4.0 для весов pyannote — оформить в NOTICE/README.
- **Взаимодействие с `redacted`-веткой не проверено.** Есть отдельный артефакт
  `redacted_transcript.txt` (`vts/api/main.py:233`). Нужно ли переносить в него
  метки `Голос N` и как это ложится на анонимизацию PII (VOS-59) — посмотреть при
  имплементации. `transcript.json` при этом никто не читает (проверено: все
  совпадения по `entries` в коде относятся к `prompt_results`, другой сущности),
  так что добавление `speaker` в него безопасно.

## Вне скоупа

- Именование спикеров, реестр голосов, merge профилей, UI-диалог → vts-80i.
- Пересборка САММАРИ с именами → vts-80i (механизм перегенерации уже существует).
- Приватность/согласия на биометрию: решение VOS-65 — пока сервис личный,
  согласия избыточны; механику вводить при переходе к мультипользовательскому
  или B2B-сценарию.
