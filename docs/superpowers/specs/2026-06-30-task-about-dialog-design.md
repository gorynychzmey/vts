# «О задаче» — диалог с параметрами запуска и результатами

**Дата:** 2026-06-30
**Статус:** проектирование (утверждён по секциям, ожидает финального ревью спека)
**Связанные задачи:** будет заведена bd-задача на имплементацию

## Проблема

В карточке задачи нет способа посмотреть, с какими параметрами она запускалась
(язык, audio_only, включён ли транскрипт, какие промпты выбраны). А сводка
результатов (общее время обработки, длины транскриптов и саммари) сейчас живёт
строкой `.task-message` между прогрессом и табами — это занимает место в карточке
и не даёт детализации по каждому промпту.

## Цель

Добавить диалог «О задаче», открывающийся по клику на кликабельный чип со
временем/размером медиа. В диалоге показать источник, параметры запуска и
результаты (включая длительность finalize-шага каждого пользовательского
промпта). Строку `.task-message` для успешного завершения убрать (её содержимое
переезжает в диалог); для failed-задач `.task-message` остаётся и показывает
только текст ошибки.

## Не-цели (YAGNI)

- Без бэкенд/API-изменений. Все нужные данные уже приходят в сериализованной
  задаче (`options`, `steps` с таймингами, `stats`, `created_at`, `source_url`).
- Без показа HTTP `headers` и без сырого JSON `options` — только осмысленные поля.
- Без отдельной кнопки-триггера: единственный вход — кликабельный чип.
- Без редактирования параметров из диалога (только просмотр).

## Архитектура

Подход **A** (единый глобальный диалог, как `presets-dialog`/`restart-final-dialog`):

- Один `<dialog id="task-about-dialog" class="tokens-dialog">` в `index.html`,
  размещён рядом с другими диалогами — **до** `<script src="app.js">` (app.js без
  `defer`, новые `getElementById` обязаны идти раньше тега script; диалоги уже
  лежат перед строкой 630 — инвариант сохраняется).
- По клику на чип: находим `.task` → его `_runtime`/`task` → `renderTaskAboutDialog(task)`
  заполняет диалог → `showModal()`.
- Закрытие: кнопка X (`#task-about-close-btn`) + клик по backdrop, как в
  `restart-final-dialog`.

Данные на фронте уже есть, поэтому правки чисто во фронтенде:
`index.html`, `app.js`, CSS, `i18n/{en,de,ru}.js`, новый UI-тест.

## Компоненты

### 1. Триггер — кликабельный чип

Текущий `.task-stats` (`index.html` ~стр. 285) — `<div>` с текстом
«1:47:49 · 212,4 MB», заполняется в `renderTaskStats` (`app.js` ~стр. 933),
скрыт пока нет ни одной метрики медиа.

Изменения:
- `.task-stats` становится чипом-кнопкой (`<button class="task-stats-chip">`):
  иконка ℹ + текстовая часть метрик.
- CSS: пилюля с фоном/рамкой, `cursor: pointer`, hover-эффект.
- Видимость не меняется: чип появляется вместе с первой метрикой медиа
  (`renderTaskStats` прячет, когда метрик нет). На queued-задаче без метрик чипа
  нет — это сознательно, текст метрик есть часть чипа.
- `renderTaskStats` переписать так, чтобы иконка ℹ была статична, а обновлялась
  только текстовая часть (сейчас пишется `textContent` целиком).
- Клик → открыть диалог для этой задачи.

### 2. Диалог «О задаче»

`#task-about-dialog`: заголовок + X + три секции.

**Секция «Источник»:**
- Название задачи (source_title или URL).
- URL/источник (для upload — имя файла из `source_url` вида `file://...`).
- Дата создания (`created_at`, локализованная).

**Секция «Параметры запуска»:**
- Язык распознавания (`options.language` или «авто», если null).
- Только аудио (`options.audio_only`: да/нет).
- Транскрипт (`options.transcript`: вкл/выкл).
- Выбранные промпты — список по именам, через `promptDisplayName`
  (из `options.prompt_results`, fallback на `options.prompts` + `selectedPromptRefs`).

**Секция «Результаты»** (видна только для `completed`; иначе скрыта):
- Общее время обработки (`stats.processing_seconds`).
- Сырой транскрипт (`stats.transcript_chars`).
- Обработанный транскрипт (`stats.redacted_chars`).
- Саммари (`stats.summary_chars`).
- Тайминги по промптам: таблица «имя промпта → длительность finalize-шага».
  Длительность = `finished_at − started_at` шага `finalizeStepName(source, id)`
  из `task.steps`. Только finalize-шаг (общий summary-head не дублируем).
  Имя промпта — через `promptDisplayName`.

Переиспользуем существующие i18n-ключи `stats.chars`/`stats.*` и форматтеры
(`formatMetricChars`, `formatMetricDuration`, `formatDuration`) где возможно.

Маппинг величин (подтверждён по `success.completed_stats` и
`_task_stats_for_serialization`):
- Rohtranskript / Сырой транскрипт = `transcript_chars`
- Verarbeitetes Transkript / Обработанный = `redacted_chars`
- Zusammenfassung / Саммари = `summary_chars`

### 3. Правка `.task-message`

- Убрать показ успешного сообщения с цифрами из карточки.
- `resolveTaskMessage` оставляет только ветку ошибки (`resolveFailureMessage`);
  ветка `resolveCompletedMessage` для карточки больше не используется.
- Логику форматирования цифр результатов переиспользовать в
  `renderTaskAboutDialog` (вынести в общий хелпер, чтобы не дублировать).
- `.task-message` в DOM остаётся (для текста ошибки failed-задач).

## Поток данных

1. Поллинг/SSE обновляют `task` и `runtime` (как сейчас).
2. `renderTaskStats` рендерит чип (иконка + метрики), управляет видимостью.
3. Клик по чипу → `renderTaskAboutDialog(task)` читает `task.options`,
   `task.steps`, `task.stats`, `task.created_at`, `task.source_url` →
   заполняет секции → `showModal()`.
4. Диалог переиспользуется для любой задачи (единственный экземпляр).

## Обработка ошибок / краевые случаи

- Задача без метрик медиа (queued) → чипа нет → диалог недоступен. Ок.
- Задача в процессе → секция «Результаты» скрыта.
- Промпт без завершённого finalize-шага → в таблице таймингов либо «—», либо
  строка отсутствует (решить при имплементации; предпочтительно «—» для
  стабильного состава строк).
- failed-задача → секция результатов скрыта; текст ошибки остаётся в
  `.task-message` в карточке.
- Отсутствующие поля `stats` (null) → существующие форматтеры уже отдают
  `stats.unknown`.

## i18n

Новые ключи в `i18n/{en,de,ru}.js`:
- заголовок диалога, подписи трёх секций;
- подписи полей: язык, только аудио, транскрипт, промпты, дата создания, источник;
- значения «авто» / «да» / «нет» / «вкл» / «выкл» (переиспользовать имеющиеся,
  если есть);
- заголовки таблицы таймингов (промпт / длительность);
- aria-label/title для чипа и кнопки закрытия (переиспользовать `tokens.close`).

## Тестирование

UI-сценарий `tests/ui/scenarios/task-about-dialog.mjs` (по образцу
`restart-dialog.mjs`):

1. closed-состояние: `#task-about-dialog` скрыт до открытия.
2. клик по `.task-stats-chip` открывает диалог (`dialog.open === true`).
3. секция параметров содержит язык и имена выбранных промптов.
4. для completed-задачи видна секция результатов, и в таблице таймингов есть
   строка(и) промптов с длительностью.
5. закрытие по X скрывает диалог.
6. отсутствие JS-ошибок на boot и при открытии.

Запуск: `cd tests/ui && node run.mjs` — все сценарии зелёные.
