# Динамическая агрегация метрик прогресса — per-step running average

**Задача:** vts-b6t (P3). Follow-up из [custom-prompts-design](2026-06-27-custom-prompts-design.md), раздел «Прогресс».

## Проблема

Веса шагов прогресс-бара захардкожены в клиенте: `STEP_WEIGHT_SECONDS` и
`FINAL_SUMMARY_WEIGHT_FALLBACK_SECONDS` в [`vts/static/app.js`](../../../vts/static/app.js#L72)
— устаревшее ручное среднее по 4 прогонам, не перемеренное после перехода на
per-prompt finalize fan-out. Числа калибруются вручную и стареют.

## Скоуп vts-b6t (этот спек)

**One-off перекалибровка по уже накопленным данным.** Берём статистику по всем
завершённым задачам (вне зависимости от пользователя), считаем актуальные веса,
**разово** обновляем константы в `app.js`. Без новой рантайм-подсистемы, без
изменения схемы БД, без нового API.

Логика агрегации выносится в переиспользуемую чистую функцию, чтобы followup
(per-user персистентные веса с периодическим пересчётом) переиспользовал её без
переписывания.

**Out of scope (followup, отдельная задача):** per-user веса в БД, периодический
пересчёт по конфиг-интервалу, seed из этих констант. См. раздел «Followup».

## Источник данных

`steps` (Postgres): `name`, `started_at`, `finished_at` (оба уже персистятся,
[`vts/db/models.py`](../../../vts/db/models.py#L107)). Число окон на задачу —
`task.summary_progress["total"]` (читается через
[`summary_progress_for_task`](../../../vts/services/task_progress.py#L7)).

**Фильтр:** только шаги со `status=completed`, принадлежащие задачам со
`status=completed` (реально завершённая работа). Длительность шага =
`finished_at - started_at` в секундах.

## Алгоритм агрегации

**Робастность: медиана**, не среднее — устойчиво к выбросам без ручных порогов.

Для каждого имени шага:
- **Фиксированные шаги** (`download`, `extract_audio`, `trim_initial_silence`,
  `segment_audio`, `detect_language`, `transcribe_segments`, `merge_transcript`,
  `prepare_llama_model`, `prepare_summary_chunks`): вес = медиана длительности
  шага по всем подходящим прогонам.
- **`summarize_windows`** масштабируется от числа окон → нормализуем **на одно
  окно**: для каждого прогона `duration / window_total` (где `window_total =
  summary_progress["total"]`, прогоны с `total < 1` пропускаются), затем медиана.
  Это совпадает с тем, как клиент применяет вес сегодня:
  [`estimateFinalSummaryWeight`](../../../vts/static/app.js#L787) делит
  `summarize_windows` на число окон. Семантика не меняется — меняется только
  источник числа.

  > Замечание по совместимости с клиентом: клиент сейчас хранит
  > `summarize_windows` как «на весь шаг» и делит на число окон в рантайме. Раз
  > мы храним «на окно», скрипт должен напечатать значение в той же шкале, что
  > клиент ожидает. Решаем при имплементации одним из двух способов, явно: либо
  > печатаем «на окно» и правим одну строку клиента, чтобы он умножал на число
  > окон, либо печатаем восстановленное «на шаг» для типичного числа окон. Это
  > единственная точка, требующая решения в коде; зафиксировать в плане.

- **`FINAL_SUMMARY_WEIGHT_FALLBACK_SECONDS`**: медиана длительности
  `summarize_final` (системный финальный вызов). При отсутствии данных —
  оставить текущее значение.

**Пустой/скудный вход:** если для шага нет данных — шаг пропускается, прежнее
значение константы остаётся (скрипт не обнуляет неизвестное).

## Архитектура

Две единицы, чёткая граница I/O ↔ чистая логика:

### 1. Чистая функция — `vts/metrics/step_weights.py`

```
aggregate_step_weights(rows: list[StepDuration]) -> dict[str, float]
```
- `StepDuration` = `(name: str, duration_sec: float, window_total: int | None)`
  (или эквивалентная структура). Никакого доступа к БД, времени, I/O.
- Возвращает `{step_name: median_seconds}`, с нормализацией на окно для
  `summarize_windows`. Шаги без данных в результат не попадают.
- Полностью детерминирована и тестируема в изоляции.

Followup переиспользует ровно эту функцию: скармливает строки с фильтром по
`user_id`, пишет результат в новую таблицу вместо печати.

### 2. Тонкий скрипт — `scripts/recompute_step_weights.py`

- Подключается к БД (через существующую конфигурацию подключения проекта).
- Запрос: completed-шаги completed-задач + `summary_progress.total` соответствующей
  задачи → строит список `StepDuration`.
- Зовёт `aggregate_step_weights`.
- Печатает готовый JS-блок (`STEP_WEIGHT_SECONDS = {...}` +
  `FINAL_SUMMARY_WEIGHT_FALLBACK_SECONDS`) для вставки в `app.js`, плюс справочно
  — число прогонов на шаг (n), чтобы видеть, где данных мало.

Скрипт не правит `app.js` сам — печатает, человек вставляет (one-off,
осознанный). Числа фиксируются коммитом в `app.js`.

## Тестирование

`tests/test_step_weights.py` — на чистую функцию:
- медиана по нескольким значениям (нечётное/чётное число точек);
- нормализация `summarize_windows` на окно (duration/total);
- прогоны с `window_total < 1` исключаются из `summarize_windows`;
- пустой вход для шага → шаг отсутствует в результате;
- выброс (один огромный прогон) не сдвигает медиану заметно.

Скрипт не тестируется юнитами (тонкая I/O-обёртка); проверяется ручным прогоном
на дампе/проде при перекалибровке.

## Изменения в коде (итог)

- **Новое:** `vts/metrics/step_weights.py`, `scripts/recompute_step_weights.py`,
  `tests/test_step_weights.py`.
- **Правка:** числа `STEP_WEIGHT_SECONDS` / `FINAL_SUMMARY_WEIGHT_FALLBACK_SECONDS`
  в `vts/static/app.js` (+ возможная одна строка нормализации, см. замечание выше).
- **Без** изменений схемы БД и без нового API.

## Followup (отдельная задача, не в этом скоупе)

Веса становятся переменными per-user, хранятся в БД, периодически
пересчитываются:
- Накопление статистики продолжается (с указанием `user_id`, если ещё не так).
- Новая таблица per-user весов; пересчёт раз в интервал (конфиг, дефолт — неделя).
- Пересчёт по умолчанию — по всем имеющимся данным пользователя; интервал
  настраивается. Если в интервале данных нет или мало — оставляем старые значения.
- При миграции/первом старте (нет данных у пользователя) — seed из текущих
  глобальных констант (результат vts-b6t).
- Клиент получает веса с сервера (endpoint) вместо хардкода.
- Переиспользует `aggregate_step_weights` из vts-b6t.
