# VOS-63 — Пользовательские промпты (Custom Prompts)

**Status:** Design approved, ready for implementation plan
**Date:** 2026-06-27
**Linear:** [VOS-63](https://linear.app/vostrikov/issue/VOS-63/dobavit-vozmozhnost-sozdavat-polzovatelskie-prompty)

## Проблема

Сегодня финальная обработка транскрипта зашита в один промпт (`global_prompt`),
результат — единственный summary. Пользователю может быть нужно применить к
транскрипту несколько разных промптов (не только «сделай саммари»), каждый со
своим результатом.

Цель: дать пользователю создавать собственные промпты и при создании задачи
выбирать набор промптов, применяемых к подготовленному транскрипту. Каждый
промпт выполняется независимо, его результат сохраняется отдельно.

## Концепция и поток обработки

Тяжёлая часть пайплайна (map-reduce по окнам) **не меняется и считается один
раз**. Меняется только финальная стадия: вместо одного `global_prompt` —
N независимых финальных вызовов.

```
аудио → сырой транскрипт
  → summarize_windows (segment_prompt по окнам)
  → [pack_prompt при необходимости]      ← подготовленный вход, считается ОДИН раз
  → для каждого выбранного промпта P:
        финальный вызов с system_prompt = P  → результат P (сохраняется отдельно)
```

- `summary` — это просто **встроенный системный промпт** `global_prompt`,
  применяемый тем же финальным вызовом, что и остальные.
- Промпты выполняются **последовательно**, **независимо** друг от друга (результат
  одного не влияет на другой).
- На каждый промпт — ровно один финальный LLM-вызов. Подготовленный вход
  (окна + pack) переиспользуется всеми.

### Два типа промптов

- **Системные** — файлы на диске (`prompts/global_prompt.md` и будущие). Описаны
  реестром. Несъёмные, не редактируются из UI. Имена локализованы (en/ru/de).
- **Пользовательские** — записи в БД, приватные на пользователя. Имя произвольное,
  не локализуется.

Оба типа попадают в общий список выбора с пометкой `system`/`user`. Имена могут
совпадать → ссылка на промпт всегда **составная** (`source` + `id`).

## Модель данных

### Таблица `prompts` (пользовательские)

```
id            UUID  PK
user_id       UUID  FK → users      (приватны на пользователя)
name          Text                  (произвольное, не локализуется)
system_prompt Text
created_at    timestamptz
updated_at    timestamptz
```

### Реестр системных промптов

Список вида:

```
{ key: "summary", file: "global_prompt.md", i18n_name_key: "prompt.system.summary" }
```

- `key` — стабильный идентификатор, он же `id` в ссылке `system:<key>`.
- Отображаемое имя берётся из i18n-словарей по `i18n_name_key` (новые ключи в
  `vts/static/i18n/*`).
- Сейчас один элемент: `summary → global_prompt.md`. Точное расположение реестра
  (конфиг настроек / манифест рядом с `prompts/`) определяется на этапе плана —
  предпочтительно рядом с существующей загрузкой промптов в `processor.py`.

### Ссылка на промпт (PromptRef)

```
{ source: "system" | "user", id: <key для system | UUID для user> }
```

Используется и в `options.prompts`, и в API, и в именах шагов/файлов результатов.

### Выбор промптов в задаче

В `task.options` под ключом `prompts`:

```json
{ "prompts": [ {"source":"system","id":"summary"}, {"source":"user","id":"<uuid>"} ] }
```

- Порядок = порядок выполнения.
- Пустой список разрешён = только транскрипт, без финальной обработки.
- Старый ключ `options.summary: bool` **удаляется** (см. «Миграция и совместимость»).

### Результаты (вариант: файлы + JSON-индекс)

- Каждый результат пишется в отдельный файл, напр.
  `summary/results/<source>__<id>.md`.
- На задаче — JSON-список `prompt_results`:

  ```json
  [ {"source":"system","id":"summary","name":"Summary",
     "path":"…/results/system__summary.md","status":"completed","progress":{…}} ]
  ```

  (новая колонка `prompt_results JSON` или эквивалент — определяется в плане).
- **Встроенный summary** продолжает писать `summary.md` и заполнять
  `task.summary_path` / `task.summary_progress` как раньше → старые задачи и
  существующий summary-флоу не ломаются.
- `prompt_results` несёт записи для **всех** выбранных промптов (включая summary),
  чтобы UI единообразно строил дропдаун результатов. Для summary `path`
  указывает на тот же `summary.md`. (Источник истины для самого summary остаётся
  `task.summary_path`; запись в `prompt_results` — индекс для UI.)

## API и MCP

> **Breaking change.** Схема создания задачи меняется в HTTP и MCP. Старый
> булев `summary` уходит. Это сознательное решение — закладываем расширяемую
> схему сразу, до накопления внешних клиентов на старом контракте.

### HTTP

**`TaskCreateRequest`** ([`vts/api/schemas.py`](../../../vts/api/schemas.py)):

- Убрать `summary: bool` (и алиас `do_summary`).
- Добавить `prompts: list[PromptRef]`, дефолт `[{source:"system", id:"summary"}]`.
- Валидация: непустой `prompts` требует `transcript=True` (обобщение текущего
  «summary requires transcript»).

**Эндпоинты промптов:**

- `GET    /api/prompts` — объединённый список **system + user**. Каждый элемент:
  `{source, id, name, editable}`. Для system `name` уже локализован под язык
  клиента. Источник и для UI-дропдаунов, и для MCP.
- `POST   /api/prompts` — создать пользовательский (`name`, `system_prompt`).
- `PATCH  /api/prompts/{id}` — редактировать пользовательский.
- `DELETE /api/prompts/{id}` — удалить пользовательский.
- (Дублирование реализуется клиентом как `POST` с предзаполненными значениями
  существующего промпта — отдельного эндпоинта не требуется.)

**Чтение результата:**

- `GET /api/tasks/{id}/results/{source}/{ref}` — единый эндпоинт чтения текста
  результата конкретного промпта. Существующий `GET …/summary`, который читает
  вкладка Summary, переводится на этот общий эндпоинт (summary = `system/summary`).

### MCP ([`vts/mcp/tools.py`](../../../vts/mcp/tools.py))

- `create_task`: `summary: bool` → `prompts: list[PromptRef]` (**breaking**).
- Новые инструменты: `list_prompts`, `create_prompt`, `update_prompt`,
  `delete_prompt`. `list_prompts` — тот же объединённый список, источник для выбора
  промптов в `create_task`.
- `get_summary` **удаляется**. Вместо него единый
  `get_prompt_result(task_id, ref)` — чтобы забрать summary, клиент передаёт
  `ref = system:summary`.
- `wait until="summary"` — без изменений (по-прежнему завязан на
  `task.summary_path`, который продолжает заполняться).

## UI

### Создание задачи

Чекбокс `#summary` ([`index.html`](../../../vts/static/index.html) ~L182) →
**мультиселект-дропдаун «Промпты»**:

- Список из `GET /api/prompts`, элементы с пометкой `system`/`user`.
- По умолчанию отмечен `system:summary`.
- Пустой выбор разрешён (= только транскрипт).
- Отправляется как `options.prompts: [ref, …]`.

### Управление промптами

Новая кнопка в верхнем меню → панель/модалка:

- Список пользовательских промптов: create / edit / delete (`name`, `system_prompt`).
- Системные промпты показываются read-only (нельзя редактировать/удалять).
- **Дублировать** — кнопка у каждого промпта (system и user). Создаёт
  пользовательский промпт с тем же `system_prompt`. Стартовое имя — копия
  (для системного берётся его локализованное имя как начальное значение),
  дальше редактируется произвольно. Это основной путь «начать свой промпт от
  системного».

### Вкладка результатов

Вкладка Summary → вкладка результатов промптов:

- Вверху дропдаун выбора результата конкретного промпта.
- Список наполняется **по мере готовности** результатов (промпты идут
  последовательно — результат появляется, как только готов).
- При выборе — текст подгружается через results-эндпоинт.
- По умолчанию показывается результат summary (если был выбран), иначе — первый
  готовый.

### Прогресс

Файл [`vts/static/app.js`](../../../vts/static/app.js) (~L41).

- **Каждый финальный промпт — отдельный шаг DAG**, а не подшаг одного
  `summarize_final`. Хвост DAG становится **динамическим**: вместо одного
  `summarize_final` — по шагу на каждый выбранный промпт.
- Имя шага несёт ссылку на промпт, чтобы UI сопоставил шаг ↔ результат:
  `finalize:system:summary`, `finalize:user:<id>`.
- Общие шаги (`prepare_llama_model`, `prepare_summary_chunks`,
  `summarize_windows`, pack) остаются и считаются один раз.
- Серверный DAG ([`vts/pipeline/types.py`](../../../vts/pipeline/types.py),
  `processor.py`) перестаёт быть статичным в хвосте — список шагов зависит от
  числа выбранных промптов.
- Вес шага финализации = текущий вес одного финального вызова (как сейчас
  оценивается `estimateFinalSummaryWeight`). Бар движется N дискретными
  ступенями.
- Константы `STEP_WEIGHT_SECONDS` разово **перекалибровать** по уже накопленным
  метрикам прогонов (one-off ручная правка чисел, без новой подсистемы метрик).

## Миграция и совместимость

- **Данные:** старые задачи имеют `options.summary: bool` и не имеют
  `options.prompts`. Чтение должно интерпретировать отсутствие `prompts`:
  `summary=true` (или отсутствие ключа) → `[{system,summary}]`; `summary=false`
  → `[]`. Встроенный summary старых задач читается из `task.summary_path` без
  изменений. Без обязательной миграции строк БД.
- **MCP/HTTP контракт:** breaking change (drop `summary`, drop `get_summary`).
- **Changelog:** в проекте нет CHANGELOG-файла; версия только в
  `vts/__init__.py`. В рамках задачи завести `CHANGELOG.md` и записать туда
  breaking change **или** явно прописать его в описании релизного тега. Решить на
  ревью спеки/плана.

## Follow-up тикеты (вне scope VOS-63)

- **Шаблоны настроек задачи** — пользователь сохраняет готовый набор опций
  (включая выбор промптов) и выбирает шаблон при создании задачи.
- **Динамическая агрегация метрик прогресса** — персистентное running-average
  хранение длительностей шагов, сервер отдаёт актуальные веса, клиент считает от
  них (вместо ручной калибровки констант).
