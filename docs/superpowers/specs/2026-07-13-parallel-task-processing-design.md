# Параллельная обработка заданий: лейны по типам шагов (VOS-85 / vts-rhs)

**Дата:** 2026-07-13
**Linear:** https://linear.app/vostrikov/issue/VOS-85/sdelat-parallelnuyu-obrabotku-zadanij
**bd:** vts-rhs

## Цель

Повысить пропускную способность очереди: сейчас воркер обрабатывает задачи строго по
одной, хотя этапы разных задач нагружают разные ресурсы (сеть, CPU/ffmpeg, GPU) и могут
перекрываться. Пока одна задача суммаризируется на GPU, другая должна скачиваться и
конвертироваться. Дополнительно: различать в статусах «вообще не приступали» (`queued`)
и «частично обработано, ждём очереди на следующий этап» (новый статус `waiting`),
и показывать позицию задачи в очереди конкретного лейна.

## Принятые решения

- **Оптимизируем пропускную способность** пачки задач, а не время отдельной задачи.
- **Один процесс воркера, asyncio-лейны.** Деплой (podman + systemd) не меняется.
  Тяжёлая работа и так внешняя: yt-dlp/ffmpeg — сабпроцессы, Whisper/LLM — HTTP-сервисы.
- **GPU — один общий лейн** для Whisper и LLM (VRAM/компьют не делим), но с
  **двухклассовым приоритетом**: asr-шаги обгоняют llm-шаги (см. ниже).
- **Скачивание — 1 слот** (YouTube троттлит параллельные загрузки), но в отдельном
  лейне: качаем параллельно с ffmpeg/GPU других задач.
- **Новый статус задачи `waiting`** — первоклассный статус в БД/API/UI, а не
  вычисляемое поле.
- Вариант «диспетчер на уровне шагов с очередями в БД» отклонён: крупный рерайт
  `processor.py` ради выгод, которые лейны дают заметно дешевле.

## Архитектура

### Пул задач в воркере

`worker_loop` ([vts/worker/main.py](../../../vts/worker/main.py)) вместо одной
`running_task` держит до `VTS_WORKER_MAX_ACTIVE_TASKS` (default 4) корутин
`processor.process_task()`. Пока в пуле есть место и в БД есть `queued`-задачи —
воркер добирает их существующим `Repo.dequeue_task()`. Лимит пула ограничивает
заодно расход диска (скачанные, но не обработанные медиа).

Cancel становится пер-задачным: наблюдение за `is_cancel_requested` для каждой
активной корутины (сейчас цикл проверяет только единственную running-задачу).
Ожидание в лейне отменяемо: cancel будит ожидающего и снимает его из очереди лейна;
pause проверяется на границах шагов, как сейчас (`_check_paused` → `TaskPaused`).

### LaneManager

Новый модуль `vts/worker/lanes.py` — очереди и слоты в памяти воркера (процесс один,
Redis для координации не нужен).

| Лейн | Слоты (default) | Шаги |
|---|---|---|
| `network` | 1 | `download` |
| `ffmpeg` | 2 | `extract_audio`, `trim_initial_silence`, `segment_audio` |
| `gpu` класс `asr` | 1 (общий на лейн) | `detect_language`, `transcribe_segments` |
| `gpu` класс `llm` | —″— | `prepare_llama_model`, `summarize_windows`, `pack_window_notes`, `finalize:*` |
| без лейна | — | `merge_transcript`, `prepare_summary_chunks` (мгновенные) |

Маппинг шаг→лейн живёт рядом с `DAG_HEAD` в
[vts/pipeline/types.py](../../../vts/pipeline/types.py); неизвестное имя шага —
ошибка конфигурации пайплайна (fail fast), кроме динамических `finalize:*` → llm.

**Приоритет в GPU-лейне.** Внутри класса — FIFO. При освобождении слота его получает
старейший ожидающий класса `asr`; если таких нет — класса `llm`. Анти-голодание:
после `VTS_GPU_ASR_BURST` (default 3) выдач `asr` подряд при непустой очереди `llm`
слот принудительно получает `llm`; любая выдача `llm` сбрасывает счётчик. Смысл
приоритета: транскрипции короче суммаризаций и «производят» работу для llm-класса —
пропуская их вперёд, держим GPU-очередь наполненной. Так как слот захватывается на
один шаг, преемпция не нужна: обгон происходит на границах шагов.

**Ночной режим** переезжает из `HeavySlot` в acquire GPU-лейна: вне разрешённого окна
GPU-слоты не выдаются (ожидающие висят в очереди лейна в статусе `waiting`).
Не-GPU лейны ночным режимом не ограничиваются — как и сейчас.

**`HeavySlot` упраздняется** ([vts/services/heavy_slot.py](../../../vts/services/heavy_slot.py)):
его лимит заменяют GPU-слоты, ночной режим переезжает в лейн. Сброс ключа
`heavy_slots` в Redis при старте воркера удаляется.

### Точка врезки — `_run_step`

В `TaskProcessor._run_step` ([vts/pipeline/processor.py](../../../vts/pipeline/processor.py)):
определить лейн шага → если свободного слота нет, перевести задачу в `waiting`
(+ SSE-событие) и встать в очередь лейна → получив слот, перевести в `running` и
выполнить шаг → освободить слот в `finally`. Код самих шагов не меняется.

Пункт реализации: **аудит `TaskProcessor` на состояние, расшаренное между задачами**.
Инстанс один на все корутины; текущие поля (`_task_metrics`, `_task_n_ctx`) — словари
по task_id, это ок, но пройтись по всем полям и по шагам на предмет глобальных
временных файлов/курсоров.

Donor-clone выполняется до шагов и лейнов не касается.

## Статусы, БД, рекавери

Жизненный цикл: `queued` (не приступали) → взята в пул → `waiting` (шаг ждёт лейн)
⇄ `running` (шаг выполняется) → `completed`. `paused`/`canceled`/`failed`/`archived` —
без изменений. Задача, попавшая в пул при свободном лейне, проходит в `running` сразу,
минуя `waiting`.

- `TaskStatus` += `waiting`; alembic-миграция `0004_task_status_waiting` по образцу
  `0003_task_status_archived` (enum не-нативный, alter мгновенный).
- `requeue_running_tasks` при старте воркера возвращает в `queued` и `running`,
  и `waiting`.
- MCP: `TaskStatusLiteral` += `waiting`; для `wait`-инструмента `waiting` —
  активное (нетерминальное) состояние.
- SSE `task_status`: добавляются переходы `waiting↔running` (~2 события на шаг,
  шагов ~12 — приемлемо).
- Позиция в лейне и имя лейна в БД **не хранятся** — это эфемерное состояние воркера;
  при рестарте пересоздаётся через requeue.

## Позиции очередей: API и UI

**Источник.** LaneManager при каждом изменении очередей пишет в Redis снапшот
`{prefix}queue:lanes` — JSON `{"network": [task_id, …], "ffmpeg": […],
"gpu_asr": […], "gpu_llm": […]}` в порядке очереди, TTL ~10 с (страховка от
умершего воркера). Это идеологическая замена текущего 2-секундного кэша глобальных
позиций: API так же читает из Redis.

**API.** В сериализацию задачи добавляется поле `queue: "network"|"ffmpeg"|"gpu"|null`
рядом с существующим `queue_position`:

- `status=queued` → как сейчас: глобальная позиция среди queued, `queue=null`;
- `status=waiting` → `queue` = лейн, `queue_position` = позиция внутри него
  (`gpu_asr`/`gpu_llm` наружу показываются одним лейном `gpu`, позиция — внутри
  своего класса).

Изменения аддитивные, старые клиенты не ломаются.

**UI** ([vts/static/app.js](../../../vts/static/app.js)):

- `queued` — как сейчас: «в очереди (№N)»;
- `waiting` — бейдж «ждёт: скачивание | конвертация | GPU (№N)», цвет как у queued;
  прогресс-бар уже выполненных шагов остаётся видимым;
- переводы ru/en/de (те же три локали, что у остальных ключей статусов).

Прогресс-бар и веса шагов не меняются — они считаются от шагов, не от статуса.

## Конфигурация и выкатка

| Env | Default | Что делает |
|---|---|---|
| `VTS_WORKER_MAX_ACTIVE_TASKS` | 4 | размер пула задач в полёте |
| `VTS_LANE_NETWORK_SLOTS` | 1 | слоты скачивания |
| `VTS_LANE_FFMPEG_SLOTS` | 2 | слоты ffmpeg |
| `VTS_LANE_GPU_SLOTS` | 1 | слоты GPU (Whisper+LLM вместе) |
| `VTS_GPU_ASR_BURST` | 3 | анти-голодание: max asr-выдач подряд при ждущем llm |

- **Kill switch:** `VTS_WORKER_MAX_ACTIVE_TASKS=1` ≈ текущее последовательное
  поведение; откат правкой env-файла на хосте, без релиза.
- Настройка `heavy_slot_limit` удаляется; из env-файла на хосте убрать
  `VTS_HEAVY_SLOT_LIMIT`, если задан. Настройки night_mode остаются как есть.
- Выкатка обычным релизом: миграция 0004, аддитивный API, бамп версии в
  `vts/__init__.py`.

## Тестирование

- **LaneManager (unit):** FIFO внутри класса; приоритет asr над llm; правило burst=3;
  отмена ожидающего снимает его из очереди; освобождение слота при исключении в шаге;
  ночной режим блокирует выдачу GPU-слота (мокаем часы).
- **Процессор/воркер (integration, с фейковыми шагами по образцу
  `tests/test_pipeline_resume.py`):** перекрытие двух задач (одна занимает gpu,
  вторая проходит download/ffmpeg); переходы `waiting↔running` и SSE-события;
  cancel в `waiting` и в `running`; pause на границе шага; рекавери возвращает
  `waiting` в `queued`.
- **API:** сериализация `queue`/`queue_position` из Redis-снапшота (waiting) и
  глобальной позиции (queued).
- **MCP:** схемы с `waiting`; `wait` не завершается на `waiting`.
- **UI:** перед тегом релиза — `verifier-web` (бейдж «ждёт: … (№N)» на карточке).

## Вне объёма

- Раздельные GPU-очереди Whisper/LLM с одновременной работой обеих моделей
  (возможный эксперимент после; конфиг слотов это позволит).
- Несколько воркер-процессов / шаги как единицы работы в БД.
- Приоритизация задач пользователем.
