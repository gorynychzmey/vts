# vts-2or — Перезапуск финальной обработки с другим набором промптов

**Status:** Design approved, ready for implementation plan
**Date:** 2026-06-28
**Beads:** vts-2or
**Follow-up:** vts-a93 (рефакторинг restart в per-task контракт — вне scope)

## Проблема

Пользователь может захотеть перегенерировать финальный результат задачи с
**другим набором промптов** — не понравилось предыдущее саммари, или нужен
иной промпт поверх уже обработанного транскрипта. Сейчас финальный перезапуск
(`restart_summary` mode `final_only`) прогоняет только тот же зашитый набор и не
даёт выбрать промпты.

Цель: финальный перезапуск может прогнать выбранный набор промптов, переиспользуя
готовый «обработанный транскрипт» (windows + pack).

## Концепция

Модель — «новая временная ветка»: при перезапуске с изменённым набором старая
история финала не сохраняется. `options.prompts` заменяется на новый набор,
выбранные промпты прогоняются **целиком** заново (не инкрементом), результаты
убранных промптов удаляются. Набор задачи и её результаты всегда консистентны.

Граница переиспользования = существующий `final_only`:

- **Переиспользуем** (не трогаем): `download` → … → `summarize_windows` →
  `pack_window_notes`. Это обработанный транскрипт; он не зависит от того, какой
  финальный промпт прогоняется.
- **Сбрасываем и считаем заново**: финальный хвост — `summarize_final` (для
  `system:summary`) + `finalize:{source}:{id}` (для каждого выбранного
  пользовательского/системного промпта).

«Целиком» означает: **все** финальные шаги нового набора сбрасываются в `pending`
и пересчитываются — включая промпты, которые остались в наборе (их прежние
результаты затираются). Это сознательно: пользователь просит перезапуск финала,
потому что предыдущий результат не устроил.

## API

Расширяем существующий `RestartSummaryRequest`
([`vts/api/schemas.py`](../../../vts/api/schemas.py)):

```python
class RestartSummaryRequest(BaseModel):
    task_ids: list[UUID] = Field(min_length=1, max_length=100)
    mode: Literal["full", "final_only"] = "full"
    prompts: list[PromptRef] | None = None   # новое; учитывается только при final_only
```

`PromptRef` — уже существующая модель (`{source: "system"|"user", id: str}`).

Поведение `POST /api/tasks/restart_summary`
([`vts/api/main.py`](../../../vts/api/main.py)):

- **`final_only` + `prompts` задан** → «новая ветка»: сменить `options.prompts`
  на присланный набор, удалить результаты убранных промптов, пересобрать
  финальный хвост шагов под новый набор, поставить задачу в очередь.
- **`final_only` + `prompts is None`** → текущее поведение (перезапуск финала тем
  же набором), без изменений.
- **`full`** → без изменений (полный пересчёт суммаризации тем же набором).

**Валидация:**

- `prompts` задан, но `mode != "final_only"` → 422 (`prompts` применим только к
  финальному перезапуску).
- `prompts == []` (пустой) при `final_only` → 422 (перезапуск финала без единого
  промпта бессмыслен; убрать всю финальную обработку — это не перезапуск).
- Каждый ref валидируется через `parse_ref` (422 на плохой ref).

**Гейт `can_restart_final_summary_task`**
([`vts/api/main.py`](../../../vts/api/main.py)): сейчас требует, чтобы
`system:summary` был в наборе. Обобщаем: финальный перезапуск разрешён, если
`summarize_windows` completed (обработанный транскрипт готов) и задача в
completed/failed-с-упавшим-финалом — **независимо** от состава набора. Требование
именно `system:summary` снимаем.

> Примечание: эта правка гейта влияет и на существующий `final_only` без
> `prompts` для задач с кастомным набором без summary — что корректно (такие
> задачи тоже должны уметь перезапускать финал). Покрыть тестом.

## Серверная механика сброса

При `final_only` + `prompts` сервер атомарно (одна транзакция + файловые
операции) приводит задачу в новое состояние:

1. **`options.prompts`** ← новый нормализованный набор. Write-back на JSON-колонку
   (паттерн `Repo.set_task_prompt_results` — reassign `task.options`, не in-place).
2. **Результаты убранных промптов.** Для каждого промпта, присутствующего в старом
   `prompt_results` (или старом наборе), но отсутствующего в новом наборе:
   - удалить файл `summary/results/{source}__{id}.md` (+ `.json`, если есть);
   - удалить его запись из `options.prompt_results`.
   - если убран `system:summary` — удалить `summary/final.md` + `summary/final.json`
     (и зеркала `outputs/summary.{md,json}`), обнулить `task.summary_path`.
   Поскольку режим «целиком», финальные артефакты **оставшихся** промптов тоже
   затираются (они будут пересчитаны) — то есть очищаются финальные результаты
   ВСЕХ промптов, а записи `prompt_results` сбрасываются под новый набор.
3. **Финальный хвост `steps` — пересобрать под новый набор** (целевой хвост =
   `build_dag_steps(new_options)` минус head):
   - удалить строки `steps` для финальных шагов, которых нет в новом хвосте
     (старые `finalize:*` убранных промптов; `summarize_final` если summary убран);
   - для каждого финального шага нового хвоста — обеспечить строку в состоянии
     `pending` (attempt=0, started_at/finished_at/message = NULL): update
     существующей или insert недостающей.
   - **Осторожно с `UniqueConstraint(task_id, name)`**: реализовать как
     update-existing + insert-missing + delete-removed, не слепой insert.
4. **Head-шаги** (`download` … `pack_window_notes`) — не трогаем (остаются
   completed, переиспользуются).
5. `summary_progress` ← `(0, 0)`; `task.status` ← `queued`; затем `notify_queued`.

Worker, разбирая задачу, пройдёт `build_dag_steps(options)`: head-шаги completed →
dry-run пропустит их; финальные шаги pending → выполнятся, переиспользуя
`packed_notes.json`/`windows.json` как готовый вход.

## UI

Меню перезапуска на карточке задачи (сейчас два пункта в
[`vts/static/index.html`](../../../vts/static/index.html)):

- **«Restart from scratch»** (`full`) — без изменений: `confirm()` → запрос тем же
  набором.
- **«Restart final summary only»** — новое поведение: вместо немедленного запроса
  открывает **диалог выбора промптов**.

**Диалог финального перезапуска:**

- Мультиселект промптов — **переиспользуем компонент с формы создания** (тот же
  рендер: системные + пользовательские с бейджами).
- Предзаполнен текущим набором задачи (`task.options.prompts`).
- Кнопка «Перезапустить» → `POST /api/tasks/restart_summary` с
  `{task_ids:[id], mode:"final_only", prompts:[...]}`, затем `loadTasks()`.
- Кнопка «Отмена» / × закрывает.
- Кнопка перезапуска заблокирована при пустом выборе (зеркалит серверный 422).
- i18n-ключи для заголовка/кнопок в `vts/static/i18n/{en,ru,de}.js`.

**Рефакторинг для переиспользования:** текущий мультиселект завязан на единственный
`#prompt-select` (id) и глобальные `renderPromptSelect`/`getSelectedPrompts`/
`syncSummaryToggle`. В диалоге перезапуска нужен **независимый экземпляр** (своя
область, свой выбор, предзаполненный из задачи). Выделить рендер мультиселекта в
переиспользуемую функцию, принимающую контейнер + начальный выбор, чтобы не
дублировать логику и не конфликтовать с селектором формы создания. Это часть
плана.

## Прогресс

Уже покрыто. Клиентский `getEnabledSteps(task)` строит хвост из
`task.options.prompts` (динамический, сделан в VOS-63 Task 13). Как только
`options.prompts` сменится при перезапуске, прогресс-бар автоматически отразит
новый финальный хвост. Отдельной работы по прогрессу не требуется.

## Тестирование

- Сброс-механика (unit/repo-уровень): смена `options.prompts`; удаление файлов и
  записей `prompt_results` убранных промптов; пересборка финального хвоста `steps`
  (update/insert/delete, без нарушения unique-ключа); head-шаги нетронуты;
  статус → queued.
- API: `final_only` + `prompts` → 200 + новое состояние; `prompts` при `full` →
  422; пустой `prompts` → 422; плохой ref → 422; гейт смягчён (custom-only задача
  без summary может перезапустить финал).
- Back-compat: `final_only` без `prompts` → прежнее поведение; `full` → без
  изменений.
- Прогон через authed-client харнесс (`tests/conftest.py`).

## Out of scope

- **vts-a93** — рефакторинг `restart_summary` в однозадачный per-task эндпоинт
  (`POST /api/tasks/{task_id}/restart_summary`, без батча `task_ids`). Отдельная
  задача.
- Изменение обработки транскрипта (windows/pack) при смене набора — не входит;
  набор влияет только на финальный хвост.
