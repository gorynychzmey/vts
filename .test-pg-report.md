# DB-integration tests on real Postgres (VOS-63)

## Summary
Migrated DB-integration test fixtures from in-memory SQLite to real Postgres
(Postgres+asyncpg, matching prod). Removed the `aiosqlite` dev dependency.

## Container engine used
`docker` on this host is podman-emulated (`Emulate Docker CLI using podman`).
Ran the throwaway Postgres via the `docker` CLI (= podman):
`docker run -d --rm --name vts-test-pg -e POSTGRES_USER=vts -e POSTGRES_PASSWORD=vts -e POSTGRES_DB=vts_test -p 5432:5432 postgres:16`

## Files changed
- `tests/_db.py` (new) — shared `make_test_engine()` + single `TEST_DATABASE_URL`
  resolution from `VTS_TEST_DATABASE_URL`, default
  `postgresql+asyncpg://vts:vts@localhost:5432/vts_test`. Created as a dedicated
  module (not inline in conftest) because the suite has two unpackaged
  `conftest.py` files (`tests/conftest.py` and `tests/mcp/conftest.py`) that
  collide under the bare `conftest` module name during full-suite collection —
  so `from conftest import ...` is not safe. `from _db import ...` is unambiguous.
- `tests/conftest.py` — `authed_app` now uses `make_test_engine()` and
  drops+recreates the schema before the test and drops it in `finally`. Seed
  user, sessionmaker (`expire_on_commit=False`), dependency overrides, and the
  `client` fixture are unchanged.
- `tests/test_prompts_repo.py` — `session` fixture now uses `make_test_engine()`
  with the same drop/create/drop lifecycle.
- `requirements-dev.txt` — removed `aiosqlite>=0.20,<1`. pytest / pytest-asyncio
  left intact.

## Fixture pattern
```python
engine = make_test_engine()  # create_async_engine(VTS_TEST_DATABASE_URL)
async with engine.begin() as conn:
    await conn.run_sync(Base.metadata.drop_all)    # clean slate
    await conn.run_sync(Base.metadata.create_all)
try:
    ... yield ...
finally:
    async with engine.begin() as conn:
        await conn.run_sync(Base.metadata.drop_all)
    await engine.dispose()
```
URL resolution lives once in `tests/_db.py`; no duplicated literal URL strings.

## Postgres-strictness findings
None. The models port cleanly to Postgres — UUID columns, JSON, and the
`native_enum=False` (VARCHAR) enums (`TaskStatus`/`StepStatus`) all round-trip.
No model changes were needed; no incompatibilities surfaced.

## Concerns
- `tests/_db.py` import relies on pytest putting `tests/` on `sys.path` (default
  rootdir/`prepend` import mode). Works here; would need an `__init__.py` /
  package layout only if the project switches to `importmode=importlib`.
- Schema drop+create per test is correct but serializes on DDL; for many DB tests
  a transaction-rollback strategy would be faster. Not needed at current scale.
- The throwaway Postgres container was cleaned up after the run.
